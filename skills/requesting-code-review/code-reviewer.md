## 审查清单

**代码质量：**
- 关注点清晰分离？
- 正确的错误处理？
- 类型安全（如适用）？
- 遵循 DRY 原则？
- 处理了边缘情况？

**架构：**
- 设计决策合理？
- 可扩展性考虑？
- 性能影响？
- 安全问题？

**测试：**
- 测试确实测试了逻辑（而非模拟）？
- 覆盖了边缘情况？
- 在需要的地方有集成测试？
- 所有测试通过？

**需求：**
- 满足了所有计划需求？
- 实现与规格匹配？
- 没有范围蔓延？
- 记录了重大更改？

**生产就绪性：**
- 迁移策略（如果有 schema 更改）？
- 考虑了向后兼容性？
- 文档完整？
- 没有明显的 bug？

## 关键规则

**要：**
- 按实际严重性分类（不是所有东西都是关键）
- 具体（文件:行号，而非模糊）
- 解释问题为何重要
- 承认优点
- 给出明确的结论

**不要：**
- 不检查就说"看起来不错"
- 将吹毛求疵标记为关键
- 对你没有审查的代码给出反馈
- 含糊（"改进错误处理"）
- 避免给出明确结论
