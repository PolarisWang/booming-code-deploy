# 文档审查系统设计

## 概述

向 booming 工作流添加两个新的审查阶段：

1. **规格文档审查** - 头脑风暴之后，writing-plans 之前
2. **计划文档审查** - writing-plans 之后，实现之前

两者都遵循实现审查使用的迭代循环模式。

## 规格文档审查者

**目的：** 验证规格是否完整、一致，并已准备好进行实现规划。

**位置：** `skills/brainstorming/spec-document-reviewer-prompt.md`

**检查内容：**

| 类别 | 要查看什么 |
|------|-----------|
| 完整性 | TODO 标记、占位符、"TBD"、未完成章节 |
| 覆盖范围 | 缺失的错误处理、边缘情况、集成点 |
| 一致性 | 内部矛盾、冲突需求 |
| 清晰性 | 模糊需求 |
| YAGNI | 未被请求的功能、过度设计 |

**输出格式：**
```
## 规格审查

**状态：** 已批准 | 发现问题

**问题（如有）：**
- [章节 X]：[问题] - [为何重要]

**建议（仅供参考）：**
- [不阻止批准的建议]
```

**审查循环：** 发现问题 -> 头脑风暴 agent 修复 -> 重新审查 -> 重复直到批准。

**派发机制：** 使用 Task 工具和 `subagent_type: general-purpose`。审查者提示模板提供完整提示。brainstorming 技能的控制器派发审查者。

## 计划文档审查者

**目的：** 验证计划是否完整、是否符合规格，以及任务分解是否合理。

**位置：** `skills/writing-plans/plan-document-reviewer-prompt.md`

**检查内容：**

| 类别 | 要查看什么 |
|------|-----------|
| 完整性 | TODO 标记、占位符、未完成的任务 |
| 规格对齐 | 计划覆盖规格需求，无范围蔓延 |
| 任务分解 | 任务原子化、边界清晰 |
| 任务语法 | 任务和步骤上的复选框语法 |
| 块大小 | 每块不超过 1000 行 |

**块定义：** 块是计划文档中任务的逻辑分组，由 `## Chunk N: <name>` 标题分隔。writing-plans 技能根据逻辑阶段创建这些边界（例如，"基础"、"核心功能"、"集成"）。每个块应足够自包含，可以独立审查。

**规格对齐验证：** 审查者同时收到：
1. 计划文档（或当前块）
2. 用于参考的规格文档路径

审查者读取两者并比较需求覆盖情况。

**输出格式：** 与规格审查者相同，但范围限于当前块。

**审查过程（逐块）：**
1. writing-plans 创建块 N
2. 控制器派发计划文档审查者，提供块 N 内容和规格路径
3. 审查者读取块和规格，返回结论
4. 如果有问题：writing-plans agent 修复块 N，转到步骤 2
5. 如果批准：继续到块 N+1
6. 重复直到所有块都被批准

**派发机制：** 与规格审查者相同——带 `subagent_type: general-purpose` 的 Task 工具。

## 更新后的工作流

```
头脑风暴 -> 规格 -> 规格审查循环 -> writing-plans -> 计划 -> 计划审查循环 -> 实现
```

**规格审查循环：**
1. 规格完成
2. 派发审查者
3. 如果有问题：修复 -> 转到 2
4. 如果批准：继续

**计划审查循环：**
1. 块 N 完成
2. 为块 N 派发审查者
3. 如果有问题：修复 -> 转到 2
4. 如果批准：下一块或实现

## Markdown 任务语法

任务和步骤使用复选框语法：

```markdown
- [ ] ### 任务 1：名称

- [ ] **步骤 1：** 描述
  - 文件：路径
  - 命令：cmd
```

## 错误处理

**审查循环终止：**
- 无硬性迭代限制——循环持续直到审查者批准
- 如果循环超过 5 次迭代，控制器应向人类提示以获得指导
- 人类可以选择：继续迭代、带已知问题批准或中止

**分歧处理：**
- 审查者是顾问——他们标记问题但不阻塞
- 如果 agent 认为审查者反馈不正确，应在其修复中解释原因
- 如果同一问题经过 3 次迭代后分歧持续，向人类提示

**格式错误的审查者输出：**
- 控制器应验证审查者输出是否有必需字段（状态、问题，如适用）
- 如果格式错误，重新派发审查者并附带有关预期格式的说明
- 2 次格式错误响应后，向人类提示
