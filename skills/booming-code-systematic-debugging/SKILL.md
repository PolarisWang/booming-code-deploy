# 系统性调试

## 概述

随机修复浪费时间并引入新 bug。快速补丁掩盖了底层问题。

**核心原则：** 在尝试修复之前，始终找到根本原因。修复症状就是失败。

**违反这个流程的字面规定就是违反了调试的精神。**

## 铁律

```
在进行根本原因调查之前，不得进行任何修复
```

如果你还没有完成第一阶段，你不能提出修复方案。

## 使用时机

适用于任何技术问题：
- 测试失败
- 生产环境中的 bug
- 意外行为
- 性能问题
- 构建失败
- 集成问题

**特别要在以下情况使用：**
- 时间紧迫时（紧急情况使猜测变得诱人）
- "只需一个快速修复"看起来显而易见时
- 你已经尝试了多次修复
- 上一次修复没有奏效
- 你不完全理解问题

**不要跳过的情况：**
- 问题看起来简单（简单的 bug 也有根本原因）
- 你在赶时间（匆忙保证返工）
- 经理想要立即修复（系统性比反复折腾更快）

## 四个阶段

在进入下一个阶段之前，你必须完成每个阶段。

### 第一阶段：根本原因调查

**在尝试任何修复之前：**

1. **仔细阅读错误信息**
   - 不要跳过错误或警告
   - 它们通常包含确切的解决方案
   - 完整地阅读堆栈跟踪
   - 记录行号、文件路径、错误代码

2. **一致地重现**
   - 你能可靠地触发它吗？
   - 确切的步骤是什么？
   - 每次都会发生吗？
   - 如果无法重现 → 收集更多数据，不要猜测

3. **检查最近的变更**
   - 什么变更可能导致这个问题？
   - Git diff、最近的提交
   - 新依赖项、配置变更
   - 环境差异

4. **在多组件系统中收集证据**

   **当系统有多个组件时（CI → 构建 → 签名，API → 服务 → 数据库）：**

   **在提出修复之前，添加诊断埋点：**
   ```
   对每个组件边界：
     - 记录进入组件的数据
     - 记录离开组件的数据
     - 验证环境/配置传播
     - 在每层检查状态

   运行一次以收集显示中断位置的证据
   然后分析证据以识别失败的组件
   然后调查该特定组件
   ```

   **示例（多层系统）：**
   ```bash
   # 第 1 层：工作流
   echo "=== Secrets available in workflow: ==="
   echo "IDENTITY: ${IDENTITY:+SET}${IDENTITY:-UNSET}"

   # 第 2 层：构建脚本
   echo "=== Env vars in build script: ==="
   env | grep IDENTITY || echo "IDENTITY not in environment"

   # 第 3 层：签名脚本
   echo "=== Keychain state: ==="
   security list-keychains
   security find-identity -v

   # 第 4 层：实际签名
   codesign --sign "$IDENTITY" --verbose=4 "$APP"
   ```

   **这揭示了：** 哪一层失败（secrets → workflow ✓，workflow → build ✗）

5. **追踪数据流**

   **当错误在调用栈深处时：**

   完整的向后追踪技术见本目录中的 `root-cause-tracing.md`。

   **快速版本：**
   - 坏值从哪里产生？
   - 是什么以坏值调用了这里？
   - 一直追踪到找到来源
   - 在来源处修复，而非症状处

### 第二阶段：模式分析

**在修复之前找到模式：**

1. **找到有效的示例**
   - 在同一代码库中找到类似的有效代码
   - 与正在出问题的代码类似的什么是有效的？

2. **与参考资料比较**
   - 如果在实现模式，完整阅读参考实现
   - 不要略读——逐行阅读
   - 在应用之前完全理解该模式

3. **识别差异**
   - 有效和损坏之间有什么不同？
   - 列出每一个差异，无论多么小
   - 不要假设"那不可能重要"

4. **理解依赖关系**
   - 这需要哪些其他组件？
   - 需要什么设置、配置、环境？
   - 它做了什么假设？

### 第三阶段：假设和测试

**科学方法：**

1. **形成单一假设**
   - 清晰地陈述："我认为 X 是根本原因，因为 Y"
   - 写下来
   - 具体，而非模糊

2. **最小化测试**
   - 做出测试假设的最小可能变更
   - 每次只改变一个变量
   - 不要同时修复多件事

3. **验证后再继续**
   - 有效吗？是 → 第四阶段
   - 无效？形成新假设
   - 不要在上面叠加更多修复

4. **当你不知道时**
   - 说"我不理解 X"
   - 不要假装知道
   - 寻求帮助
   - 深入研究

### 第四阶段：实现

**修复根本原因，而非症状：**

1. **创建失败测试用例**
   - 最简单的可能重现
   - 如果可能，使用自动化测试
   - 如果没有框架，使用一次性测试脚本
   - 修复前必须有
   - 使用 `booming-code:test-driven-development` 技能编写正确的失败测试

2. **实现单一修复**
   - 解决已识别的根本原因
   - 每次一个变更
   - 不要"顺便"改进
   - 不要捆绑重构

3. **验证修复**
   - 测试现在通过了吗？
   - 没有其他测试损坏？
   - 问题实际上解决了吗？

4. **如果修复无效**
   - 停止
   - 计数：你尝试了多少次修复？
   - 如果 < 3：返回第一阶段，用新信息重新分析
   - **如果 ≥ 3：停止并质疑架构（下面的步骤 5）**
   - 在没有架构讨论的情况下不要尝试第 4 次修复

5. **如果 3 次以上修复失败：质疑架构**

   **表明架构问题的模式：**
   - 每次修复在不同地方揭示新的共享状态/耦合/问题
   - 修复需要"大规模重构"才能实现
   - 每次修复在其他地方产生新症状

   **停下来质疑基本原则：**
   - 这个模式从根本上是合理的吗？
   - 我们是否"凭借惯性坚持下去"？
   - 应该重构架构还是继续修复症状？

   **在尝试更多修复之前与 your human partner 讨论**

   这不是失败的假设——这是错误的架构。

## 红旗——停下来并遵循流程

如果你发现自己在想：
- "先快速修复，之后再调查"
- "就试着改 X 看看是否有效"
- "添加多个更改，运行测试"
- "跳过测试，我会手动验证"
- "可能是 X，让我修复那个"
- "我不完全理解，但这可能有效"
- "模式说 X，但我会以不同方式适应它"
- "以下是主要问题：[列出修复而无调查]"
- 在追踪数据流之前提出解决方案
- **"再尝试一次修复"（已经尝试了 2 次以上）**
- **每次修复在不同地方揭示新问题**

**所有这些都意味着：停止。返回第一阶段。**

**如果 3 次以上修复失败：** 质疑架构（见第四阶段第 5 步）

## your human partner 表明你做错了的信号

**注意这些重定向：**
- "那没有发生吗？" - 你在没有验证的情况下做了假设
- "会告诉我们...吗？" - 你应该添加证据收集
- "停止猜测" - 你在没有理解的情况下提出修复
- "超级思考这个" - 质疑基本原则，而非只是症状
- "我们卡住了？"（沮丧）- 你的方法不起作用

**当你看到这些时：** 停止。返回第一阶段。

## 常见合理化

| 借口 | 现实 |
|------|------|
| "问题很简单，不需要流程" | 简单问题也有根本原因。流程对简单 bug 很快。 |
| "紧急，没有时间走流程" | 系统性调试比猜测和检查的反复折腾更快。 |
| "先试试这个，然后调查" | 第一次修复设定了模式。从一开始就做对。 |
| "我会在确认修复有效后再写测试" | 未经测试的修复不持久。先测试证明它有效。 |
| "一次多个修复节省时间" | 无法隔离什么起了作用。导致新 bug。 |
| "参考太长了，我来适应该模式" | 不完整的理解保证产生 bug。完整阅读它。 |
| "我看到了问题，让我修复它" | 看到症状 ≠ 理解根本原因。 |
| "再尝试一次修复"（在 2+ 次失败后） | 3+ 次失败 = 架构问题。质疑模式，不要再修复。 |

## 快速参考

| 阶段 | 关键活动 | 成功标准 |
|------|---------|---------|
| **1. 根本原因** | 阅读错误、重现、检查变更、收集证据 | 理解什么和为什么 |
| **2. 模式** | 找到有效示例，比较 | 识别差异 |
| **3. 假设** | 形成理论，最小化测试 | 确认或新假设 |
| **4. 实现** | 创建测试，修复，验证 | bug 解决，测试通过 |

## 当流程揭示"没有根本原因"时

如果系统性调查揭示问题确实是环境性的、时序依赖或外部原因：

1. 你已完成了流程
2. 记录你调查了什么
3. 实现适当的处理（重试、超时、错误消息）
4. 添加监控/日志以供未来调查

**但是：** 95% 的"没有根本原因"情况是调查不完整。

## 支持技术

这些技术是系统性调试的一部分，可在本目录中找到：

- **`root-cause-tracing.md`** - 通过调用栈向后追踪 bug 以找到原始触发器
- **`defense-in-depth.md`** - 找到根本原因后在多层添加验证
- **`condition-based-waiting.md`** - 用条件轮询替换任意超时

**相关技能：**
- **booming-code:test-driven-development** - 用于创建失败测试用例（第四阶段，步骤 1）
- **booming-code:verification-before-completion** - 在声明成功之前验证修复有效

## 实际影响

来自调试会话：
- 系统性方法：15-30 分钟修复
- 随机修复方法：2-3 小时的反复折腾
- 首次修复成功率：95% vs 40%
- 引入的新 bug：接近零 vs 常见
