# 活动执行守卫

## 概述

只要仓库中存在 `docs/executions/CURRENT.md`，就意味着当前有一个活动计划。活动计划在被处理之前，禁止进入任何新的回答、解释、规划、实现、文档修改或知识整理。

**核心原则：** `docs/executions/CURRENT.md` 存在 = 有活动计划。

## 使用时机

在以下任一情况出现时，先运行本技能：

- 你即将回答用户问题
- 你即将提出澄清问题
- 你即将开始规划、实现、测试、调试或写文档
- 你不知道仓库里是否存在活动计划

## 流程

### 第一步：检查活动计划

1. 检查 `docs/executions/CURRENT.md` 是否存在。
2. 如果不存在：退出本技能，继续正常的技能发现与响应流程。
3. 如果存在：先读取该文件，提取以下信息：
   - 计划名称
   - 当前状态
   - 最近执行摘要
   - 下一步

### 第二步：阻止其他响应

在活动计划被处理之前，不要回答用户原本的问题。只允许提示用户先处理当前计划。

使用类似下面的格式：

```text
检测到当前正在执行的计划：
- 计划：<计划名称>
- 状态：<当前状态>
- 最近摘要：<最近执行摘要>
- 下一步：<下一步>

请选择：
1. 继续
2. 放弃
```

### 第三步：处理用户选择

**如果用户选择 `继续`：**

- 基于 `CURRENT.md` 中的结构化上下文恢复执行。
- 先回到当前计划，不要先回答不相关问题。
- 继续时应遵循计划中指定的执行路径：`executing-plans` 或 `subagent-driven-development`。

**如果用户选择 `放弃`：**

- 将 `CURRENT.md` 归档到 `docs/executions/history/`
- 归档状态必须是 `abandoned`
- 清空当前位，让仓库回到“无活动计划”状态
- 只有在归档完成后，才能继续新的对话或任务

## 放弃与完成的边界

- 本技能**不提供** `完成` 选项
- `completed` 只能由执行技能在以下条件同时满足时自动归档：
  - 计划任务全部完成
  - 必要验证全部通过
  - 本轮应写入的 wiki 更新已完成

## CURRENT.md 的最低要求

恢复执行前，`CURRENT.md` 至少应包含：

- 设计文档路径
- 计划文档路径
- 当前状态
- 任务总数与当前进度
- 已确认的重要上下文
- 用户已确认决策
- 当前约束
- 风险与阻塞
- 最近执行摘要
- 下一步

如果它不完整，也不能绕过。它仍然代表一个活动计划，必须先处理。

## 红旗

**绝不：**

- 在检测到 `CURRENT.md` 后先回答其他问题
- 提供 `完成` 作为用户可选项
- 因为 `CURRENT.md` 内容不完整就忽略它
- 把 `CURRENT.md` 当成普通笔记而不是活动计划状态
- 在未归档 `abandoned` 之前开始新任务

## 集成

**由以下技能优先调用：**

- `dev:using-booming` - 任何响应前的全局入口

**协作技能：**

- `dev:executing-plans` - 维护 `CURRENT.md` 与 completed archive
- `dev:subagent-driven-development` - 维护 `CURRENT.md` 与 completed archive
