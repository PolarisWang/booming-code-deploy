## 指令优先级

Booming 技能覆盖默认系统提示行为，但**用户指令始终优先**：

1. **用户的明确指令**（CLAUDE.md、GEMINI.md、AGENTS.md、直接请求）——最高优先级
2. **Booming 技能**——在冲突时覆盖默认系统行为
3. **默认系统提示**——最低优先级

## 如何访问技能

**在 Claude Code 中：** 使用 `Skill` 工具。当你调用技能时，其内容会被加载并呈现给你——直接遵循它。永远不要用 Read 工具读取技能文件。

**在 Gemini CLI 中：** 技能通过 `activate_skill` 工具激活。Gemini 在会话开始时加载技能元数据，并按需激活完整内容。

**在其他环境中：** 查看你的平台文档了解技能如何加载。

## 规则

**在任何响应或行动之前调用相关或被请求的技能。**

**不要再依赖独立的活动任务守卫技能。** 如果你即将进入新的计划执行流程，由对应执行技能自己处理 `docs/dev/ACTIVE.md`、恢复当前任务，或提示 `继续 / 挂起 / 放弃`。

**不要把活动任务检查误用成每次都拦截用户的总开关。** 对当前会话中的小修、小查、小验证、局部样式/文案调整、日志查看和低风险支持动作，直接处理，不要无谓打断用户。

## 技能优先级

1. **流程技能优先**（头脑风暴、调试）——这些决定如何处理任务
2. **规划与执行技能其次**（`roadmap`、`writing-plans`、`executing-plans`、`subagent-driven-development`）——这些决定如何分解、恢复和推进主线
3. **实现技能最后**（frontend-design、mcp-builder 等）——这些指导具体执行

## 工作流规则

- "让我们构建 X" → 先 `brainstorming`，再根据分流结果进入 `roadmap` 或 `writing-plans`
- "修复这个 bug" → 先 `systematic-debugging`，再进入领域技能
- "实现一个已有计划" → `executing-plans` 或 `subagent-driven-development`

## 红旗

以下想法意味着你在合理化：

- "这只是个小问题，不需要技能"
- "我先看几眼代码再说"
- "我先开始新的复杂流程，稍后再处理活动任务"
- "我记得技能要求，不用再看"

## 用户指令

指令说的是什么，而不是如何。"添加 X" 或 "修复 Y" 不意味着跳过工作流。
